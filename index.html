CORE REASONING LOOP

Every ARIA Action Follows Four Stages

A standardized four-stage reasoning loop ensures consistency, auditability, and alignment with RSM methodology. The human always approves the final action.

STAGE 1 — PULL

Pull Work Item Context

Retrieve the work item from Cosmos DB: title, description, discussions, attachments, metadata, and linked SVD decisions.

STAGE 2 — REFERENCE

Reference Best Practices

Query Microsoft Learn MCP for D365 best practices relevant to the task type. Cross-reference RSM embedded methodology from the prompt library.

STAGE 3 — CLARIFY

Clarify Context

Ask exactly three targeted clarifying questions using RAG against the full Cosmos DB project context. Questions constrained to what is genuinely unknown.

STAGE 4 — PROPOSE

Propose Action

Combine best practices with project-specific context and present options. User reviews and approves the proposed action before any MCP call modifies any system.

GUARDRAILS & SCOPE CONTROLS

Purpose-Built, Methodology-Constrained

ARIA is not a general-purpose AI assistant. It operates within tightly controlled boundaries at every layer, enforced automatically without depending on user discipline.

Task-Scoped Sessions

ARIA is bound to a specific Engagement Hub task. No general-purpose chat mode, freeform assistant, or conversational AI outside D365 implementation work.

Four-Stage Reasoning Constraint

Every interaction follows Pull → Reference → Clarify → Propose. Off-topic inputs are not actionable within this loop and are rejected gracefully.

Human-in-the-Loop Approval

No MCP call that modifies any system executes without explicit user approval. Every action presented as a structured card with methodology reference.

Scoped MCP Actions

D365 configuration limited to the Vibe/VibeConfigStarterKit instruction set. No arbitrary commands against client environments.

Build Retry Limits

Autonomous code generation includes hard retry limits (3-5 attempts). Beyond threshold, ARIA stops and escalates to a human developer.

Tenant Isolation & Secrets

No client data crosses tenant boundaries. All credentials stored in Azure Key Vault — never in code, environment variables, or prompts.

Full Audit Trail

Every interaction logged with timestamp, user ID, project ID, task ID, and outcome. Immutable logs retained for minimum 2 years.

Methodology Traceability

Every ARIA recommendation traceable to a specific RSM Velocity step. If unmappable to methodology, it is not surfaced.

D365 F&O DEVELOPMENT PIPELINE

Autonomous D365 F&O Development

An end-to-end agent workflow that autonomously creates D365 artifacts — data schemas, business logic, and UI components — and pushes them through build, review, and deployment stages.

STAGE 01 — AGENT CONTEXT VIA MCP

Agent Gathers Context via MCP

Agent queries three MCP sources: PLP Postgres (project lifecycle data), Client Code Postgres (existing D365 codebase), and Custom D365 MCP (RSM coding best practices).

STAGE 02 — CREATE D365 F&O ARTIFACTS

Agent Creates D365 F&O Artifacts

Agent determines which layers are needed and generates artifacts across four categories:

Data Schema Tables, Views, EDTs, Base Enums
Business Logic X++ classes, CoC extensions, event handlers
UI Components Forms, Menu Items, Tiles, Workspaces
Misc Security privileges/duties, labels, resources
STAGE 03 — BUILD & RETRY ON AZURE VMs

Build on Azure VMs

Code compiled on Azure-hosted D365 build VMs. On failure, agent auto-analyzes errors and retries (3-5 max). On max retries exceeded: escalates to human developer.

ARCHITECTURAL DECISIONS

Key Decisions Made

Abandon 80+ bespoke agents

Build ARIA as a centralized, generic agent with curated prompts mapped to RSM Velocity actions. 10-week compressed delivery timeline.

Engagement Hub remains source of truth

Associates' and consultants' day-to-day workflows do not change. ARIA reads from and acts upon EH and DevOps data.

Cosmos DB as performance layer

Project data aggregated from ADO, EH, SOWs, SharePoint, and SVD content. Delta sync on every user login. Vector search for RAG.

ARIA as button in Engagement Hub

Solves multi-tenant auth, inherits project context at launch, and creates seamless design-to-implementation transition.

Auditability as first-class layer

Every action logged with full metadata to Cosmos DB. Immutable, 2-year retention. Reconstructable agent decision chains.

Model flexibility & routing

Not hard-coupled to a single LLM. Reasoning tasks route to Claude Sonnet 4 or o3; fast tasks to GPT-4o-mini. Provider swap requires no frontend changes.

SVD-gated access escalation

Individual-level access pre-SVD, global project access post-SVD. Enforced through the Engagement Hub session context.

Microsoft overlap strategy

Continue independent ARIA development. Interface abstraction layer ensures ARIA remains the UI even if underlying config logic swaps to Microsoft's solution.
